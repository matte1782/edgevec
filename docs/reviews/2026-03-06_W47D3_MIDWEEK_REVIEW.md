### AV7: Early-Stop Convergence Logic -- PASS

The `kmeans()` function (lines 573-659) implements two early-stop checks:

1. **No assignment change** (line 601): `if !changed { break; }` -- correct, stops when no vector changes cluster
2. **Max centroid movement** (lines 637-655): computes squared L2 distance between old and new centroids for all `ksub` centroids, breaks if `max_movement_sq < convergence_threshold^2` -- correct

The threshold comparison `max_movement_sq < convergence_threshold * convergence_threshold` (line 653) correctly uses squared distance vs squared threshold, avoiding a `sqrt()` call in the inner loop. This is the standard approach.

The threshold value `1e-4` is hardcoded at the call site (line 405), not exposed as a public API parameter. The plan (W47.3b) called for `convergence_threshold: f32` as a parameter on `PqCodebook::train()`. See M1 below.

**Verdict: PASS** (logic is correct)

### M1 Detail: convergence_threshold Not Exposed

**Location:** `src/quantization/product.rs` line 405

**Evidence:** W47.3b acceptance criteria states: "Add `convergence_threshold: f32` parameter to `PqCodebook::train()` (default: 1e-4)." The actual implementation hardcodes `1e-4` inside `train()` and passes it to `kmeans()`. The public API signature remains `train(vectors, m, ksub, max_iters)`.

**Impact:** Users cannot tune convergence behavior. For the immediate goal (G4 optimization), this is functionally correct since `1e-4` is the intended default. However, the acceptance criterion explicitly requires the parameter.

**Required Action:** Either (a) add `convergence_threshold` as an optional parameter to `PqCodebook::train()`, or (b) formally document that this is deferred (e.g., to a `PqConfig` builder in v0.10.0 per PQ_GO_NOGO_DECISION.md recommendation) and update the plan's acceptance criterion.
